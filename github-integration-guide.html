🚀 GitHub 集成操作指南

📌 提示：本指南将帮助您快速完成 GitHub 与您的应用程序的集成，让您可以轻松管理代码仓库和协作开发。

一、准备工作

步骤 1：注册 GitHub 账号

如果您还没有 GitHub 账号，请访问 github.com 注册一个免费账号。

步骤 2：创建个人访问令牌（Personal Access Token）

  1. 登录 GitHub 后，点击右上角头像 → Settings（设置）
  2. 在左侧菜单中找到 Developer settings（开发者设置）
  3. 选择 Personal access tokens → Tokens (classic)
  4. 点击 Generate new token（生成新令牌）
  5. 给令牌起个名字，例如：我的应用集成
  6. 选择需要的权限（至少需要 repo 权限）
  7. 点击生成，请立即复制并保存令牌（只会显示一次！）
⚠️ 重要提醒：个人访问令牌相当于您的密码，请妥善保管，不要分享给他人或提交到公开的代码仓库中。

二、集成步骤

步骤 1：配置 GitHub 集成

在您的应用程序中找到 GitHub 集成设置页面，通常在：

  • 设置 → 集成 → GitHub
  • 或者：工具 → 版本控制 → GitHub

步骤 2：输入认证信息

在集成页面中输入以下信息：

  • GitHub 用户名：您的 GitHub 账号用户名
  • 访问令牌：刚才生成的个人访问令牌
  • 默认分支：通常是 main 或 master

步骤 3：测试连接

点击"测试连接"按钮，确保配置正确。如果显示"连接成功"，说明配置正确。

三、具体操作示例

示例 1：克隆远程仓库

场景：您想要将 GitHub 上的项目下载到本地进行开发。

仓库地址：https://github.com/username/my-project.git

操作步骤：
1. 在应用中选择"克隆仓库"
2. 输入仓库地址
3. 选择本地保存路径
4. 点击"克隆"按钮

结果：项目代码将被下载到您指定的本地文件夹。

示例 2：创建新仓库

场景：您有一个本地项目，想要上传到 GitHub。

项目名称：my-awesome-app
项目描述：一个很棒的应用程序

操作步骤：
1. 在应用中选择"创建新仓库"
2. 填写仓库名称：my-awesome-app
3. 添加描述（可选）
4. 选择公开或私有
5. 点击"创建"按钮

结果：GitHub 上将创建一个新的仓库，并自动与本地项目关联。

示例 3：提交代码更改

场景：您修改了代码，想要保存到 GitHub。

修改的文件：
- src/index.js (修改了登录功能)
- styles/main.css (更新了样式)

操作步骤：
1. 在应用中查看"更改"列表
2. 选择要提交的文件
3. 输入提交信息：
   "修复登录功能并更新界面样式"
4. 点击"提交"按钮
5. 点击"推送"将更改上传到 GitHub

结果：您的代码更改将被保存并上传到 GitHub 仓库。

示例 4：创建分支进行功能开发

场景：您要开发一个新功能，但不想影响主分支。

新功能：添加用户评论功能
分支名称：feature/user-comments

操作步骤：
1. 在应用中选择"分支管理"
2. 点击"创建新分支"
3. 输入分支名称：feature/user-comments
4. 选择基于 main 分支创建
5. 切换到新分支进行开发

结果：您可以在新分支上安全地开发新功能，不会影响主分支的稳定性。

示例 5：合并代码（Pull Request）

场景：新功能开发完成，需要合并到主分支。

源分支：feature/user-comments
目标分支：main

操作步骤：
1. 确保所有更改已提交并推送
2. 在应用中选择"创建合并请求"
3. 选择源分支和目标分支
4. 填写合并请求标题和描述：
   标题："添加用户评论功能"
   描述："实现了用户可以在文章下方发表评论的功能"
5. 提交合并请求
6. 等待代码审查（如果需要）
7. 合并到主分支

结果：新功能将被合并到主分支，所有人都可以使用。

四、常见问题解决

问题 1：认证失败

症状：提示"认证失败"或"无效的令牌"

解决方案：

  • 检查个人访问令牌是否正确复制（没有多余的空格）
  • 确认令牌没有过期
  • 验证令牌具有所需的权限

问题 2：推送失败

症状：提示"推送被拒绝"

解决方案：

  • 先拉取（pull）最新代码，解决冲突后再推送
  • 检查是否有写入权限
  • 确认分支保护规则是否允许直接推送

问题 3：合并冲突

症状：提示"存在合并冲突"

解决方案：

  • 在应用中查看冲突文件
  • 手动编辑解决冲突
  • 标记冲突已解决
  • 提交合并结果

五、最佳实践建议

✅ 推荐做法

  • 频繁提交：每完成一个小功能就提交，不要积累太多更改
  • 清晰的提交信息：用简洁明了的语言描述做了什么更改
  • 使用分支：新功能和修复都在独立分支上进行
  • 代码审查：重要更改提交前请同事审查
  • 定期同步：经常拉取最新代码，避免大量冲突

提交信息模板

类型: 简短描述

详细说明（可选）

示例：
feat: 添加用户登录功能
fix: 修复页面加载错误
docs: 更新 README 文档
style: 优化代码格式
refactor: 重构支付模块
